## What is modeled

The spec covers the connection lifecycle between peers:

1. **Handshake** â€” `version` / `verack` exchange, with non-deterministic rejection (`reject`) to model version validation failures and self-connection detection.
2. **Keepalive** â€” `ping` / `pong` with nonce echo, triggered when a connection is idle.
3. **Block sync** â€” `inv` â†’ `getheaders` â†’ `headers` â†’ `getdata` â†’ `block`, looping until the lagging peer catches up.
4. **Disconnection** â€” peers disconnect if no message is received within `DisconnectTimeout` ticks, then restart the handshake.

Each connection is modeled as an explicit state machine from the perspective of peer **n** tracking its relationship with peer **m**. Any non-`init` state transitions back to `init` on a `Disconnect` (idle timeout), and `ping`/`pong` can fire from any post-handshake state when the connection is idle.

```mermaid
stateDiagram-v2
    direction LR

    [*] --> init

    init --> version_sent : version

    state "Handshake" as hs {
        version_sent --> established : verack
        version_sent --> init : reject
    }

    state "Block sync (n lags m)" as sync {
        established --> inv_sent : inv
        inv_sent --> synced : already caught up
        inv_sent --> getheaders_sent : getheaders
        getheaders_sent --> headers_sent : headers
        headers_sent --> getdata_sent : getdata
        getdata_sent --> block_received : block (still behind)
        block_received --> getdata_sent : getdata
        getdata_sent --> synced : block (caught up)
    }
```

The block sync states are only entered when n has fewer blocks than m. Once n catches up (`synced`), the session for that direction is complete â€” m independently goes through the same states from its own perspective if it also lags n.

The spec checks:
- **Liveness** â€” `AllSynced`: eventually all peers reach the same block height.
- **Safety invariants** from ZIP-0204:
  - `InvCountBounded` / `GetDataCountBounded` â€” inventory vectors carry â‰¤ 50,000 entries.
  - `HeadersCountBounded` â€” headers messages carry â‰¤ 160 headers.
  - `VersionBounded` â€” peers advertise a version â‰Ą `MinPeerProtoVersion` (170002).
  - `PingOnEstablished` â€” ping nonces are only active after the handshake completes.
  - `SyncDirection` â€” a peer only enters sync states when it actually has fewer blocks than its partner.

## Running the model checker

Requires Java and [`tla2tools.jar`](https://github.com/tlaplus/tlaplus/releases/latest).

Two configurations are provided:

| Config | Peers | Symmetry | Verification |
|---|---|---|---|
| [`protocol.cfg`](protocol.cfg) | 2 | No | Complete â€” fully explores the state space |
| [`protocol_3peers.cfg`](protocol_3peers.cfg) | 3 | Yes | Bounded â€” runs until timeout, no errors found |

```bash
# Complete liveness proof (2 peers, finishes in ~3 minutes)

# Bounded stress test (3 peers, run until timeout)
java -jar tla2tools.jar -config protocol_3peers.cfg protocol.tla
```

### Symmetry reduction

Permuting peer names produces structurally identical states â€” `peer1={1 block}, peer2={3 blocks}` is the same scenario as `peer1={3 blocks}, peer2={1 block}`. Declaring `SYMMETRY Permutations(InitialPeers)` tells TLC to collapse those equivalence classes, reducing the state space by up to N! (6x for 3 peers).

For symmetry to work, peers must be declared as **model values** (abstract atoms) rather than strings in the config:
```
CONSTANT peer1 = peer1   \* model value
```

**Important caveat:** symmetry reduction is sound for safety properties but can theoretically miss liveness counterexamples (a known TLC limitation). The 2-peer config intentionally omits symmetry to give a complete, trustworthy proof of the `AllSynced` liveness property.
